48 ### <a name="introduction">Introduction</a>
49 
50 
51 `fuzzylite` is a free and open-source fuzzy logic control library programmed in C++ for multiple platforms (e.g., Windows, Linux, Mac, iOS). [`jfuzzylite`](https://github.com/fuzzylite/jfuzzylite/) is the equivalent library for Java and Android platforms. Together, they are the FuzzyLite Libraries for Fuzzy Logic Control.
52 
53 <center>
54 ```
55 The goal of the FuzzyLite Libraries is to easily design and efficiently operate fuzzy logic controllers following an object-oriented programming model without relying on external libraries.
56 ```
57 </center>

77 ### <a name="features">Features</a>
78 
79 **(6) Controllers**: Mamdani, Takagi-Sugeno, Larsen, Tsukamoto, Inverse Tsukamoto, Hybrids
80 
81 **(21) Linguistic terms**: (4) *Basic*: triangle, trapezoid, rectangle, discrete.
82 (9) *Extended*: bell, cosine, gaussian, gaussian product, pi-shape, sigmoid difference, sigmoid product, spike.
83 (5) *Edges*: binary, concave, ramp, sigmoid, s-shape, z-shape.
84 (3) *Functions*: constant, linear, function.
85 
86 **(7) Activation methods**: general, proportional, threshold, first, last, lowest, highest.
87 
88 **(8) Conjunction and Implication (T-Norms)**: minimum, algebraic product, bounded difference, drastic product, einstein product, hamacher product, nilpotent minimum, function.
89 
90 **(10) Disjunction and Aggregation (S-Norms)**: maximum, algebraic sum, bounded sum, drastic sum, einstein sum, hamacher sum, nilpotent maximum, normalized sum, unbounded sum, function.
91 
92 **(7) Defuzzifiers**: (5) *Integral*: centroid, bisector, smallest of maximum, largest of maximum, mean of maximum.
93 (2) *Weighted*: weighted average, weighted sum.
94 
95 **(7) Hedges**: any, not, extremely, seldom, somewhat, very, function.
96 
97 **(3) Importers**: FuzzyLite Language `fll`, Fuzzy Inference System `fis`, Fuzzy Control Language `fcl`.
98 
99 **(7) Exporters**: `C++`, `Java`, FuzzyLite Language `fll`, FuzzyLite Dataset `fld`, `R` script, Fuzzy Inference System `fis`, Fuzzy Control Language `fcl`.
100 
101 **(30+) Examples** of Mamdani, Takagi-Sugeno, Tsukamoto, and Hybrid controllers from `fuzzylite`, Octave, and Matlab, each included in the following formats: `C++`, `Java`, `fll`, `fld`, `R`, `fis`, and `fcl`.

108 ### <a name="example">Example</a>
109 #### FuzzyLite Language
110 ```yaml
111 #File: ObstacleAvoidance.fll
112 Engine: ObstacleAvoidance
113 InputVariable: obstacle
114  enabled: true
115  range: 0.000 1.000
116  lock-range: false
117  term: left Ramp 1.000 0.000
118  term: right Ramp 0.000 1.000
119 OutputVariable: mSteer
120  enabled: true
121  range: 0.000 1.000
122  lock-range: false
123  aggregation: Maximum
124  defuzzifier: Centroid 100
125  default: nan
126  lock-previous: false
127  term: left Ramp 1.000 0.000
128  term: right Ramp 0.000 1.000
129 RuleBlock: mamdani
130  enabled: true
131  conjunction: none
132  disjunction: none
133  implication: AlgebraicProduct
134  activation: General
135  rule: if obstacle is left then mSteer is right
136  rule: if obstacle is right then mSteer is left
137 ```
138 ```cpp
139 //File: ObstacleAvoidance.cpp
140 #include "fl/Headers.h"
141 
142 int main(int argc, char* argv[]){
143  using namespace fl;
144  Engine* engine = FllImporter().fromFile("ObstacleAvoidance.fll");
145 
146  std::string status;
147  if (not engine->isReady(&status))
148  throw Exception("[engine error] engine is not ready:\n" + status, FL_AT);
149 
150  InputVariable* obstacle = engine->getInputVariable("obstacle");
151  OutputVariable* steer = engine->getOutputVariable("mSteer");
152 
153  for (int i = 0; i <= 50; ++i){
154  scalar location = obstacle->getMinimum() + i * (obstacle->range() / 50);
155  obstacle->setValue(location);
156  engine->process();
157  FL_LOG("obstacle.input = " << Op::str(location) <<
158  " => " << "steer.output = " << Op::str(steer->getValue()));
159  }
160 }
161 ```
162 
163 #### C++
164 ```cpp
165 //File: ObstacleAvoidance.cpp
166 #include "fl/Headers.h"
167 
168 int main(int argc, char* argv[]){
169  using namespace fl;
170  //Code automatically generated with fuzzylite 6.0.
171 
172  using namespace fl;
173 
174  Engine* engine = new Engine;
175  engine->setName("ObstacleAvoidance");
176  engine->setDescription("");
177 
178  InputVariable* obstacle = new InputVariable;
179  obstacle->setName("obstacle");
180  obstacle->setDescription("");
181  obstacle->setEnabled(true);
182  obstacle->setRange(0.000, 1.000);
183  obstacle->setLockValueInRange(false);
184  obstacle->addTerm(new Ramp("left", 1.000, 0.000));
185  obstacle->addTerm(new Ramp("right", 0.000, 1.000));
186  engine->addInputVariable(obstacle);
187 
188  OutputVariable* mSteer = new OutputVariable;
189  mSteer->setName("mSteer");
190  mSteer->setDescription("");
191  mSteer->setEnabled(true);
192  mSteer->setRange(0.000, 1.000);
193  mSteer->setLockValueInRange(false);
194  mSteer->setAggregation(new Maximum);
195  mSteer->setDefuzzifier(new Centroid(100));
196  mSteer->setDefaultValue(fl::nan);
197  mSteer->setLockPreviousValue(false);
198  mSteer->addTerm(new Ramp("left", 1.000, 0.000));
199  mSteer->addTerm(new Ramp("right", 0.000, 1.000));
200  engine->addOutputVariable(mSteer);
201 
202  RuleBlock* mamdani = new RuleBlock;
203  mamdani->setName("mamdani");
204  mamdani->setDescription("");
205  mamdani->setEnabled(true);
206  mamdani->setConjunction(fl::null);
207  mamdani->setDisjunction(fl::null);
208  mamdani->setImplication(new AlgebraicProduct);
209  mamdani->setActivation(new General);
210  mamdani->addRule(Rule::parse("if obstacle is left then mSteer is right", engine));
211  mamdani->addRule(Rule::parse("if obstacle is right then mSteer is left", engine));
212  engine->addRuleBlock(mamdani);
213 
214  std::string status;
215  if (not engine->isReady(&status))
216  throw Exception("[engine error] engine is not ready:\n" + status, FL_AT);
217 
218  for (int i = 0; i <= 50; ++i){
219  scalar location = obstacle->getMinimum() + i * (obstacle->range() / 50);
220  obstacle->setValue(location);
221  engine->process();
222  FL_LOG("obstacle.input = " << Op::str(location) <<
223  " => " << "steer.output = " << Op::str(steer->getValue()));
224  }
225 }
226 ```

228 ### <a name="compile-build-execute">Compile, Link, and Execute</a>
229 
230 Once you have an engine written in C++, you can compile it to create an executable file which links to the `fuzzylite` library. The linking can be either static or dynamic. Basically, the differences between static and dynamic linking are the following. **Static linking** includes the `fuzzylite` library into your executable file, hence increasing its size, but the executable no longer needs to have access to the `fuzzylite` library files. **Dynamic linking** does not include the `fuzzylite` library into your executable file, hence reducing its size, but the executable needs to have access to the `fuzzylite` shared library file. When using dynamic linking, make sure that the shared library files are either in the same directory as the executable, or are reachable via environmental variables:
231 
232 ```posh
233 rem Windows:
234 set PATH="\path\to\fuzzylite\release\bin;%PATH%"
235 ```
236 ```bash
237 #Unix:
238 export LD_LIBRARY_PATH="/path/to/fuzzylite/release/bin/:$LD_LIBRARY_PATH"
239 ```
240 
241 #### Windows
242 The commands to compile your engine in Windows are the following:
243 
244 C++11 (default)
245 ```posh
246 rem static linking:
247 cl.exe ObstacleAvoidance.cpp fuzzylite-static.lib /Ipath/to/fuzzylite /EHsc /MD
248 ```
249 ```posh
250 rem dynamic linking:
251 cl.exe ObstacleAvoidance.cpp fuzzylite.lib /Ipath/to/fuzzylite /DFL_IMPORT_LIBRARY /EHsc /MD
252 ```
253 
254 C++98
255 ```posh
256 rem static linking:
257 cl.exe ObstacleAvoidance.cpp fuzzylite-static.lib /Ipath/to/fuzzylite /DFL_CPP98=ON /EHsc /MD
258 ```
259 ```posh
260 rem dynamic linking:
261 cl.exe ObstacleAvoidance.cpp fuzzylite.lib /Ipath/to/fuzzylite /DFL_IMPORT_LIBRARY /DFL_CPP98=ON /EHsc /MD
262 ```
263 
264 #### Unix
265 The commands to compile your engine in Unix are the following:
266 
267 C++11 (default)
268 ```bash
269 #static linking
270 g++ ObstacleAvoidance.cpp -o ObstacleAvoidance -I/path/to/fuzzylite -L/path/to/fuzzylite/release/bin -lfuzzylite-static --std=c++11
271 ```
272 ```bash
273 #dynamic linking
274 g++ ObstacleAvoidance.cpp -o ObstacleAvoidance -I/path/to/fuzzylite -L/path/to/fuzzylite/release/bin -lfuzzylite -Wno-non-literal-null-conversion
275 ```
276 
277 C++98
278 ```bash
279 #static linking
280 g++ ObstacleAvoidance.cpp -o ObstacleAvoidance -I/path/to/fuzzylite -L/path/to/fuzzylite/release/bin -lfuzzylite-static -DFL_CPP98=ON
281 ```
282 ```bash
283 #dynamic linking
284 g++ ObstacleAvoidance.cpp -o ObstacleAvoidance -I/path/to/fuzzylite -L/path/to/fuzzylite/release/bin -lfuzzylite -DFL_CPP98=ON -Wno-non-literal-null-conversion
285 ```

295 ### <a name="building">Building from Source</a>
296 You can build the `fuzzylite` library from source using `CMake` [(cmake.org)](https://cmake.org/).
297 
298 The files [`fuzzylite/build.bat`](/fuzzylite/build.bat) and [`fuzzylite/build.sh`](/fuzzylite/build.sh) are build scripts for the Windows and Unix platforms, respectively.
299 After building from source, the resulting binaries will be located in the sub-folders `fuzzylite/release/bin` and `fuzzylite/debug/bin`. The usage of these scripts is presented as follows.
300 
301 #### Windows
302 ```bash
303 > build.bat help
304 Usage: build.bat [options]
305 where [options] can be any of the following:
306  all builds fuzzylite in debug and release mode (default)
307  debug builds fuzzylite in debug mode
308  release builds fuzzylite in release mode
309  clean erases previous builds
310  help shows this information
311 ```
312 
313 #### Unix
314 ```bash
315 $ ./build.sh help
316 Usage: [bash] ./build.sh [options]
317 where [options] can be any of the following:
318  all builds fuzzylite in debug and release mode (default)
319  debug builds fuzzylite in debug mode
320  release builds fuzzylite in release mode
321  clean erases previous builds
322  help shows this information
323 ```

327 #### Building Options
328 For advanced building options, please check the contents of [`fuzzylite/build.bat`](/fuzzylite/build.bat) or [`fuzzylite/build.sh`](/fuzzylite/build.sh), and the contents of [`fuzzylite/CMakeLists.txt`](/fuzzylite/CMakeLists.txt).
329 
330 The following building options available:
331 
332 * `-DFL_USE_FLOAT=ON` builds the binaries utilizing the `fl::scalar` data type as a `float` represented in 4 bytes. By default, the binaries are built utilizing `-DFL_USE_FLOAT=OFF` to utilize `fl::scalar` as a `double` represented in 8 bytes and hence providing better accuracy. If `fuzzylite` is built with `-DFL_USE_FLOAT=ON`, then the applications linking to `fuzzylite` also need to specify this compilation flag.
333 
334 
335 * `-DFL_CPP98=ON` builds binaries utilizing `C++98` features. By default, `fuzzylite` is built with `-DFL_CPP98=OFF` to utilize `C++11` features. If compiling for `C++98`, be aware that you will not be able to benchmark the performance of your engine using the `Benchmark` class.
336 
337 
338 * `-DFL_BACKTRACE=OFF` disables the backtrace information in case of errors (default is ON). In Windows, the backtrace information requires the external library `dbghelp`, which is generally available in your system.
339 
340 * `-DCMAKE_BUILD_TYPE=[Debug|Release]` sets the mode of your build. You can only build one mode at a time with a single CMake script.

372 
373 #### Console
374 The console application of `fuzzylite` allows you to import and export your engines. Its usage can be obtained executing the console binary. In addition, the console can be set in interactive mode. The `FuzzyLite Interactive Console` allows you to evaluate a given controller by manually providing the input values. The interactive console is triggered by specifying an input file and an output format. For example, to interact with the `ObstacleAvoidance` controller, the interactive console is launched as follows:
375 
376 ```bash
377 fuzzylite -i ObstacleAvoidance.fll -of fld
378 ```

384 ### <a name="whatsnew">What's New?</a>
385 * The FuzzyLite Libraries, namely fuzzylite and jfuzzylite, both in version 6.0, are dual licensed under the GNU General Public License version 3 and the Apache License version 2.
386 
387 * By default, fuzzylite builds using C++11 instead of C++98.
388 
389 * Important performance improvements.
390 
391 * Refactored the following names for the operation of engines: from activation operator to implication operator, from accumulation operator to aggregation operator.
392 
393 * Renamed the term `Accumulated` to `Aggregated`.
394 
395 * New activation methods decouple the activation of rules from the rule block and provide different methods for activating rules (see Activation Methods).
396 
397 * New class `ActivationFactory` provides a factory of activation methods.
398 
399 * New class `Benchmark` to evaluate the performance and accuracy of engines.
400 
401 * New class `Complexity` to estimate the computational complexity of an engine.
402 
403 * New class `RScriptExporter` to export the surfaces of an engine using the `ggplot2` library.
404 
405 * New class `Binary` term for binary edges.
406 
407 * New `UnboundedSum` S-Norm in `SNormFactory`.
408 
409 * New classes `SNormFunction` and `TNormFunction` to create custom functions on any two values using the `Function` class.
410 
411 * Added description strings to `Engine`, `Variable` and `RuleBlock`
412 
413 * Privatized previously protected members of classes and subclasses of `Term`, `Variable`, `Rule`, `Defuzzifier`, `[Cloning|Construction]Factory`, `Importer`, `Exporter`, amongst others.
414 
415 * Improved portability by replacing `int` for `std::size_t` where necessary, thereby additionally removing warnings in Windows 64bit
416 
417 * Deleted `Operation.cpp` and inlined its methods into `Operation.h`
418 
419 * Updated `.travis.yml` to use Docker, and build using g++ (versions 6, 5, 4.9, 4.8, 4.7) and clang (versions 3.8, 3.7, 3.6, and 3.5).
420 
421 * Added `appveyor.yml` to use continuous integration in Windows under Visual Studio 2013 and 2015.
422 
423 * Added some unit tests and support for future unit tests.
424 
425 * Bug fixes.
426 
427 * New example of hybrid engines.
428 
429 * New example on obstacle avoidance for Mamdani, Takagi-Sugeno, and Hybrid engines.
430 
431 * New R scripts for each example and its respective surfaces in `pdf` formats.
432 
433 #### Bug fixes
434 * Fixed bug in `CloningFactory::deregisterObject()`. Bug: Object was deleted before removing it from the map, leaving an invalid object in the map which would cause a segmentation fault. Solution: Remove the object from the map before deleting it.
435 * Fixed bug causing segmentation fault when malformed term in FuzzyLite Language
436 * Fixed bug computing the `NormalizedSum` S-Norm.
437 * Fixed bug in `RuleBlock` to reset and clone the implication operator. Bug: implication operator is not copied and reset. Fix: copy and reset implication operator when cloning the `RuleBlock`.
438 * Fixed bug in `Function` term. Bug: given a formula = "tan(y)" and a map["y"] = 1.0, and executing `Function::load(formula)`, then the map of variables is reset because `load()` calls `unload()` first, causing the deregistration of variable `y`. Solution: Removed method `unload()` from `load()`, thereby causing future `load()` not to reset variables.
439 * Fixed bug in `Function` when enclosing variable in double parenthesis.
